# Filelay 真实 iCloud 验收清单

## 目标
- 验证真实 iCloud Drive 环境下的双向同步、冲突处理、删除传播、重启恢复和大文件表现。
- 区分“本地共享目录模拟通过”和“真实 iCloud 最终行为通过”。

## 前置条件
- 两台登录同一 Apple ID 的 macOS 设备，分别记为设备 A 和设备 B。
- 两台设备都安装最新构建，并使用同一套 `Filelay` 配置格式。
- 两台设备的 iCloud Drive 已开启，且 `桌面与文稿文件夹` 是否开启状态保持一致。
- 两台设备都清空旧的测试文件与旧的 `Filelay` 测试目录，避免历史元数据干扰。

## 基础链路
- [ ] 设备 A 选择本地文件上传后，iCloud 管理区生成云端文件和 `.filelay/<cloudFileId>.json` 元数据。
- [ ] 设备 B 在没有手动关联前，不会自动创建本地文件，也不会自动建立同步关系。
- [ ] 设备 B 手动关联同内容本地文件后，状态直接进入“已同步”。
- [ ] 设备 B 手动关联不同内容本地文件后，状态进入“冲突”而不是静默覆盖。

## 自动同步
- [ ] 设备 A 修改已关联本地文件后，无需手动刷新，云端文件在可接受时间内自动更新。
- [ ] 设备 B 已关联同一云端文件时，无需手动刷新，本地文件在可接受时间内自动更新。
- [ ] 设备 B 删除本地文件后，若云端仍存在，系统会自动恢复本地文件。
- [ ] 设备 A 与设备 B 同时修改同一文件时，至少一端会进入冲突处理，不会静默覆盖。

## 删除传播
- [ ] 设备 A 删除云端文件后，iCloud 中的目标文件被移除。
- [ ] 删除后云端 sidecar 元数据保留 tombstone 信息，不会被其他设备误传回云端。
- [ ] 设备 B 下一轮同步后，关联关系被移除，但本地文件仍保留。

## 重启与恢复
- [ ] 设备 A 退出并重新启动后，已关联文件、历史、设备状态都能恢复。
- [ ] 设备 B 在离线修改本地文件后重新联网，系统能正确上传或进入冲突。
- [ ] iCloud 同步延迟较大时，系统最终仍能收敛到正确状态，不会出现无限来回覆盖。

## 大文件与边界
- [ ] 至少验证 1 个 100MB 以上文件的上传、关联和再次修改。
- [ ] 至少验证 1 个二进制文件和 1 个纯文本文件。
- [ ] 验证文件名包含空格、中文和较长扩展名时，路径与元数据保持正确。
- [ ] 验证多个文件位于同一云端子目录时，各自元数据互不污染。

## 诊断与日志
- [ ] 检查应用支持目录下的 `Logs/sync.log.jsonl` 是否持续写入结构化日志。
- [ ] 发生冲突、删除、错误时，日志里能看到对应 `category`、`cloudFileId`、`localPath`、`cloudPath`。
- [ ] 人工抽查 1 次日志和 UI 历史，确认两边事件能互相对上。

## 验收结论
- [ ] 所有基础链路通过
- [ ] 所有自动同步通过
- [ ] 删除传播通过
- [ ] 重启与恢复通过
- [ ] 大文件与边界通过
- [ ] 日志与诊断通过
